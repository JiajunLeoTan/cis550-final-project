\section{Introduction}
Axiom is a data-driven shopping assistant built on Amazon product, category, and
review data. The application helps users search products, compare alternatives,
inspect review quality signals, estimate cart savings, and explore category and
brand-level analytics. The project emphasizes a normalized relational design,
clear API contracts, and a visible original-versus-optimized query mode exposed
through the \texttt{?optimized=1} flag.

\subsection{Problem Statement}
In the modern e-commerce landscape, consumers are frequently overwhelmed by information overload. With datasets reaching millions of products and reviews, users struggle to cross-reference price volatility, brand reputation, and sentiment signals effectively. Existing platforms often lack the analytical tools necessary to consolidate these fragmented data points, preventing shoppers from identifying true market value or comparing brand performance across high-volume categories.

\subsection{Project Goal}
The goal of Axiom is to develop a high-performance, full-stack shopping to provide:
\begin{itemize}
    \item \textbf{High-Efficiency Discovery:} Real-time search and filtering across millions of rows using optimized indexing.
    \item \textbf{Analytical Insights:} Comparative brand and category metrics visualized through custom data structures.
    \item \textbf{Personalized Utility:} A customizable value-ranking engine that allows users to weight product attributes (price, stars, review depth, and recency) according to their specific priorities.
\end{itemize}

\section{Architecture}
The system has three layers. The frontend is a React/Vite single-page
application with pages for home, browsing, product detail, cart, analytics, and
value rankings. The backend is an Express server with route modules for product
workflows, cart savings, metadata, and analytics. PostgreSQL on AWS RDS stores
the five normalized relations and is accessed through the Node \texttt{pg}
connection pool.

\section{Data}
The raw data consists of \texttt{amazon\_products.csv},
\texttt{amazon\_reviews.csv}, and \texttt{amazon\_categories.csv}. The cleaner
deduplicates products by ASIN, parses numeric fields, maps category IDs,
extracts and canonicalizes brands from product titles, and keeps only reviews
whose ASIN or parent ASIN can be linked to a cleaned product. Brand entity
resolution lowercases and strips punctuation for grouping, removes common legal
suffixes such as Inc., Co., LLC, and Ltd., and drops brands with fewer than five
products so rare brand links become \texttt{NULL brand\_id}.

pair, not additional sampling inside the cleaner.

\section{Database}
\begin{center}
\includegraphics[width=0.95\linewidth]{er_diagram.png}
\end{center}

The schema has five relations: Categories, Brands, Products, Users, and Reviews.
Products reference Categories and Brands; Reviews reference Products and Users.
All tables are in third normal form:

\textbf{Categories.} The functional dependency is
\texttt{category\_id -> category\_name}; \texttt{category\_id} is the primary
key, and \texttt{category\_name} is also unique. Every non-trivial dependency has
a superkey on the left.

\textbf{Brands.} The dependency is \texttt{brand\_id -> brand\_name};
\texttt{brand\_id} is the primary key, and \texttt{brand\_name} is unique.
Therefore no non-key attribute determines another non-key attribute.

\textbf{Products.} The dependency is \texttt{asin -> title, img\_url,
product\_url, price, list\_price, stars, review\_count, is\_best\_seller,
category\_id, brand\_id}. ASIN is the primary key, so every non-trivial
dependency has a superkey determinant.

\textbf{Users.} The only stored attribute is \texttt{user\_id}, which is the
primary key. There are no non-key attributes and no transitive dependencies.

\textbf{Reviews.} The dependency is \texttt{review\_id -> asin, user\_id,
rating, review\_title, review\_text, helpful\_vote, verified\_purchase,
review\_timestamp}. \texttt{review\_id} is the primary key; product and user
details remain in their own relations, avoiding transitive dependencies.

\section{Queries}
The most complex query families are brand performance, review trends, trending
products, top value products, and value rankings. They combine joins,
aggregation, filtering, ranking, and date windows. The original variants are
kept for timing comparisons, and the optimized variants reduce repeated scans
where the planner benefits, push predicates earlier, introduce limits where the
UI only needs top results, and benefit from final delivery indexes.

Full SQL listings are in Appendix A via the route query modules:
\texttt{server/queries/analytics.sql.js}, \texttt{server/queries/products/},
and \texttt{server/queries/cart.sql.js}.

\section{Performance Evaluation}
The benchmark runner executes one discarded warmup run, waits one second, then
collects five measured \texttt{EXPLAIN ANALYZE} runs. The generated table is
saved to \texttt{docs/timings.md}. \texttt{pg\_trgm} search support remains
available, but the largest measured improvements in this run come from a
partial discount expression index and materialized views for category
comparison, brand performance, value components, and trending components. The
trending and default value-ranking optimized paths use a data-relative 12-month
review window from the materialized view, because the loaded review corpus ends
in 2023 while the server clock is in 2026. Top-value uses the requested
\texttt{reviewedSince} date against each product's latest linked review
timestamp.

\section{Technical Challenges}
The largest data challenge was aligning review ASINs with the product corpus.
The cleaner now supports \texttt{parent\_asin} fallback, but a high-volume review
table still requires a matching product source. On the backend, the main
engineering constraint was preserving the public API and original SQL variants
while adding optimized paths, validation, route-level caching, and benchmark
instrumentation.
